# Seed and screenshots pipeline

## Context

The dashboard screenshots referenced from the README (`docs/dashboard-status.png`,
`dashboard-hosts.png`, `dashboard-settings.png`) were taken months ago against
the pre-redesign UI. After the design-system migration and the security
hardening (icons, logo, paleta, encryption, must-change-password enforcement),
those PNGs no longer represent the product accurately. Re-capturing manually
every time something visual changes is fragile.

Two coupled problems to solve:

1. **No seed**: the dev SQLite starts empty or with whatever the operator
   created during testing. Screenshots taken against an empty dashboard are
   useless ("Never" everywhere); screenshots taken after running the e2e
   suite have arbitrary host names like `e2e-add-1777832535757.example.com`.
2. **No reproducible capture path**: the only screenshots flow that exists is
   the temporary script I wrote during T005, which I deleted afterwards.

This plan introduces both pieces, mirroring the pattern nudge uses (a unified
seed feeding a Playwright capture script, orchestrated by a single `make`
target). Building on top of this enables the future static landing page (out
of scope for this plan).

## Decisions confirmed with user

| Topic | Decision |
|-------|----------|
| Hostnames in the seed | `*.example.com` (RFC 2606, never registrable) |
| Public IP in the State row | `1.2.3.4` (public-looking but obviously fictitious) |
| Admin user `must_change_password` | `False` — screenshots land on the dashboard, not the change-password page |
| Seed reset behaviour | Destructive **only** when invoked with `--reset`. Without the flag, seed refuses to overwrite existing data. |
| Generated PNGs | `make screenshots` writes to `docs/dashboard-*.png`; the operator reviews and commits manually. No CI auto-commit. |
| Landing page stack | Astro + Tailwind (matches nudge's `site/`). **Deferred** — separate plan after this one lands. |
| Stack location | Seed in `scripts/seed.py` (project-level), screenshots in `e2e/screenshots.mjs`, orchestration via root `Makefile` (mirrors nudge's layout). |
| Test location | `src/test/test_seed.py` (single pytest entry point covers everything). |
| Seed access from dev container | Mount `scripts/` read-only at `/scripts` via `dev/docker-compose.yaml`, so the seed runs as `python /scripts/seed.py`. |

## Design proposal

### Seed (`scripts/seed.py`)

Plain Python script (no framework wrapping — the project uses FastAPI, not
Django, so there is no `manage.py` host). Imports `SqliteRepository`,
`init_db`, `hash_password` and the secrets helpers, then writes a curated
fixture set in a single transaction-ish run (each repository call commits its
own session per project convention).

**Default behaviour (no args)**: refuses to run if `hosts`, `state` or
`history` already have rows. Surfaces a clear error with the recovery hint:

```
Refusing to seed: database already contains data. Re-run with --reset to wipe
and re-seed, or remove data/dyndns.db first.
```

**`--reset` flag**: deletes every row in `hosts`, `history`, `state`,
`settings`, `users` (in dependency order, no FK violations) and then seeds.
Does NOT delete `data/.jwt_secret` or `data/.encryption_key` — those are
generated by the runtime, not by the seed.

**Fixture content**:

- **Admin** — `admin` / `admin` (the dev-default), `must_change_password=False`.
  Hash via the existing `hash_password()`.
- **State row** — `current_ip = "1.2.3.4"`, `last_check` = now (UTC). Fictitious
  but valid IPv4, public-looking.
- **Settings** — defaults (`update_interval=300`, `logger_level=INFO`).
- **Hosts** (mix of health states):
  - `home.example.com` — `last_status=True`, `last_update=` ~5 min ago. Healthy.
  - `vpn.example.com` — `last_status=True`, `last_update=` ~10 min ago. Healthy.
  - `files.example.com` — `last_status=True`, `last_update=` ~1 h ago. Healthy.
  - `nas.example.com` — `last_status=False`, `last_update=` ~30 min ago,
    `last_error="Authentication failed - check username/password"`. Red.
  - `media.example.com` — `last_status=None`, `last_update=None`. Pending (amber).
- **History** — ~25 entries spread across the last 7 days. Mix of:
  - 4× `ip_changed` events with the IP transitioning across realistic
    private→public values, ending at `1.2.3.4`.
  - ~12× `host_updated` (success).
  - 4× `host_failed` with realistic OVH error strings (`HTTP 401: Unauthorized`,
    `Hostname not found - check DynHost configuration in OVH`, `Authentication
    failed - check username/password`, `OVH service error - try again later`).
  - 2× `host_created`, 2× `host_deleted` (referencing a fictitious cleanup),
    1× `settings_updated`.
- The history dataset is enough to populate **two pages** of the paginated
  history view (limit 20), so screenshots can show the pagination control as
  active.

**Cifrado**: each `repo.create_host(hostname, username, password)` runs through
`encrypt_password()` automatically (T008 wired this in). The seed needs the
encryption key available at runtime — same env-var-or-file precedence as the
app. The Makefile target sets a deterministic `ENCRYPTION_KEY` for the seed
run so screenshot regeneration is reproducible without polluting `data/`.

**Determinism**: timestamps are computed relative to `now()`, so two
consecutive seed runs produce slightly different `last_update` values. That
is acceptable — it only affects "5 minutes ago" wording in the screenshot,
not the visual layout. We do NOT pin a fixed clock; the screenshots may
re-render with slightly different relative dates each refresh, which is honest.

### Screenshots (`e2e/screenshots.mjs`)

Playwright script ESM, lives in `e2e/` next to the existing test suite so it
reuses the `ovh-dyndns-e2e` Docker image (Playwright + Chromium baked in).

**Scenes captured**:

| File | What | URL / interaction |
|---|---|---|
| `dashboard-status.png` | Status section with mixed-health hosts | `/`, login, default landing |
| `dashboard-hosts.png` | Hosts management table | nav click → Hosts |
| `dashboard-history.png` | History with hostname filter dropdown collapsed | nav click → History |
| `dashboard-settings.png` | Settings form (with the version footer visible) | nav click → Settings |

**Optional scenes** (decision left for implementation, low cost): `login.png`
(initial state), `hosts-modal.png` (Add Host dialog open). Adding them later
is trivial — the script is just a list of steps.

**Viewport**: `1280×800` (fits desktop preview wide enough for the table to
show all columns without horizontal scroll). Single device pixel ratio.

**Output**: writes to `docs/dashboard-*.png` directly, overwriting the
existing stale PNGs. No suffix versioning — git history is the version log.

### Make target (`Makefile` at project root)

Following nudge's layout, introduce a project-root `Makefile` with the
`screenshots` target. The existing `dev/Makefile` stays — we do not move the
dev shortcuts. The new top-level Makefile contains *only* the orchestration
targets that span multiple folders.

**Target sequence**:

```makefile
screenshots:
    docker compose -f dev/docker-compose.yaml up -d
    docker compose -f dev/docker-compose.yaml exec --workdir /app -e ENCRYPTION_KEY=$(SEED_ENCRYPTION_KEY) ovh-dyndns-dev python /app/../scripts/seed.py --reset
    docker compose -f dev/docker-compose.yaml exec --workdir /app -d -e ENCRYPTION_KEY=$(SEED_ENCRYPTION_KEY) ovh-dyndns-dev python main.py
    @scripts/wait-for-health.sh
    docker run --rm --network host \
        -v $(CURDIR)/e2e/screenshots.mjs:/e2e/screenshots.mjs \
        -v $(CURDIR)/docs:/screenshots \
        ovh-dyndns-e2e node screenshots.mjs
    docker compose -f dev/docker-compose.yaml exec ovh-dyndns-dev sh -c "pkill -f 'python main.py' 2>/dev/null; true"
    @echo "Updated PNGs under docs/. Review and commit manually."
```

(Real Makefile will use proper escaping; the shell script wait-for-health.sh
polls `/health` with a timeout — Makefile native commands don't loop well.)

**Important**: the operator ALWAYS reviews the diff and commits manually.
`make screenshots` does not stage or commit on its own.

### Tests for the seed

A small unit test `test/test_seed.py` that:

- Imports `scripts/seed.py` as a module and runs `seed(reset=False)` against
  an empty temp DB. Asserts the expected row counts (5 hosts, 1 state, 25
  history entries, 1 admin, 1 settings).
- Runs it twice without `--reset` and asserts the second call raises
  `SystemExit` (or an explicit refusal exception) and does not duplicate rows.
- Runs `seed(reset=True)` after the first seed — second seed lands clean.
- The admin row has `must_change_password=False`.
- Spot-check that one host has `last_status=False` and a non-null `last_error`
  (the failure-state visual coverage).

The screenshots script itself is **not** unit-tested — it is a developer tool,
not production code. Its DoD is "PNGs land on disk and look right by eye".

## Scope

### What is included

- `scripts/seed.py` — destructive-with-flag fixture loader that writes 5
  hosts, ~25 history entries, an admin with the must-change flag off, a
  state row pointing to `1.2.3.4`, and default settings.
- `e2e/screenshots.mjs` — Playwright ESM script that captures the four
  canonical dashboard PNGs at 1280×800.
- `scripts/wait-for-health.sh` — tiny polling helper for the make target.
- `Makefile` at project root with a single `screenshots` target that
  orchestrates: dev container up → seed --reset → app start → wait for
  health → capture → app stop.
- `test/test_seed.py` — unit coverage for the seed (counts, idempotence
  refusal, reset round-trip, admin flag).
- README hero and per-section screenshots (the existing
  `docs/dashboard-*.png`) regenerated in this PR by running `make
  screenshots` and reviewing the diff. No README copy changes.

### What is NOT included

- The Astro + Tailwind landing page (separate plan after this one lands).
- Mobile screenshots / responsive previews (single 1280×800 viewport for
  now; mobile re-capture lives behind a future `make screenshots-mobile`
  if we ever need it).
- CI auto-regeneration on merge to main. Operator-driven for now.
- Migration of dev shortcuts from `dev/Makefile` to the project-root
  `Makefile` — those stay where they are.
- A canonical "demo password" different from `admin/admin` — the seed
  intentionally uses the dev-default so it matches what `init_admin_user`
  would create on a fresh boot.
- Pinning timestamps in fixtures. Relative-to-now is good enough.

## Implementation order

1. **Seed first.** Write `scripts/seed.py` with the fixture set. Run by hand
   (`docker compose ... exec ... python /app/../scripts/seed.py --reset`)
   and verify the dashboard manually.
2. **Test the seed.** Add `test/test_seed.py` covering the 4–5 cases from
   the design. Confirm `pytest --cov-fail-under=90` still passes.
3. **Screenshots script.** Write `e2e/screenshots.mjs` reusing the helpers
   already established (login flow, `localStorage` priming if needed). Run
   it manually against the seeded dev container, inspect the PNGs visually.
4. **wait-for-health helper.** Tiny shell script that polls `/health` with
   a timeout, exits non-zero on timeout. Used by the Makefile.
5. **Makefile.** New project-root `Makefile` with the `screenshots` target
   wiring it all together. Test the full one-shot path.
6. **Regenerate the README screenshots.** Run `make screenshots`, review
   the diff visually (open all four PNGs, eyeball them), commit them
   alongside the script changes.
7. **Validation pass.** Open `README.md` rendered (locally or on a draft
   PR) and confirm the new screenshots flow nicely with the surrounding
   prose. No copy changes expected, but proportions might need a tiny
   tweak in the `<img width="380">` attributes if the new captures are
   wider/taller than the originals.

## Risks and considerations

- **Encryption key management for the seed run**: the seed encrypts host
  passwords via `encrypt_password()`, which lazily reads the encryption key.
  If we don't pre-set `ENCRYPTION_KEY` via env in the Makefile, the seed
  generates a fresh `data/.encryption_key` and overwrites whatever was
  there. That breaks any developer that was relying on a specific key. The
  Makefile mitigates by passing a deterministic `SEED_ENCRYPTION_KEY` for
  the duration of the screenshots run. Document this in the seed's
  docstring.
- **Pre-commit hook**: the project has a `scripts/pre-commit` that runs
  `ruff check` and `ruff format --check`. The seed is Python and must
  pass both. The screenshots script is JavaScript — out of ruff's scope.
- **Path conventions inside Docker**: the dev container mounts `src/` to
  `/app` and the project root sits at `/app/..`. The seed therefore lives
  at `/app/../scripts/seed.py` from the container's POV, which is awkward.
  Two clean options to discuss: bind-mount `scripts/` into the container
  at `/scripts`, or shell into a repo-root path (`docker compose ... exec
  -w / ovh-dyndns-dev python /scripts/seed.py`). I propose the first —
  add a volume mount in `dev/docker-compose.yaml`.
- **PNG size in git**: each screenshot is 30–60 KB. Four PNGs in the repo
  = ~200 KB. Fine. `pngquant` post-processing is optional — skip for now.
- **Test coverage of seed**: `scripts/seed.py` is not under the `src/`
  tree that the coverage gate scans. It will not move the gate up or
  down. Tests for it are pure correctness checks, not coverage rescue.
  Important to keep the seed at `scripts/` and not under `src/` so it
  cannot be imported by accident from production code.
- **Operator manual review of PNGs**: explicit non-goal of CI auto-commit.
  Document in the README's contributing section (out of scope for THIS
  plan but worth noting for the next plan).
- **Time-relative fixtures**: `last_update = now - 5min` produces slightly
  different output every screenshot run. Locale on the test container
  (`toLocaleString()`) might also shift. Acceptable noise; no pinning.
